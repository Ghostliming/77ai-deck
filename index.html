CRM · 客户困境

很多 CRM 项目
卡在四个问题上

01

员工不愿意录系统——开电脑、找字段、补记录太麻烦

02

过程信息沉淀不下来——沟通、会议、跟进发生在系统外

03

系统不给一线下一步建议——只堆记录，不推动动作

04

变成管理者监督工具——员工抵触大，落地效果差

AI 要进入业务发生的场景——不是在事后补表，而是在沟通、跟进、会议发生时理解、沉淀，并推动下一步。

从「管理者监督工具」→「一线也能用得上的业务助手」

CRM · Demo recap

演示归口 · 四个痛点怎么解

痛点 ① 不爱录 → 像发消息一样录入

邮件、沟通、纪要进入 CRM，AI 自动整理为客户、商机、任务。

痛点 ② 过程沉不下去 → 业务发生时留下信息

不是事后补结果字段——跟进过程进入同一业务上下文。

痛点 ③ 不给下一步 → 识别信号、推动作

从记录中读出承诺、风险与下一步——推送任务与建议，不是单点问答。

痛点 ④ 员工抵触 → 赋能一线，不是管更严

优秀打法变成团队可用的教练与工作流——帮普通员工做判断。

信任边界：AI 给出建议、依据与风险提示；改资料、发邮件、改阶段等关键动作仍由人确认。
四痛点 · 一条业务链 · 一个边界

Insight · Question

一个常见的经营追问

经营管理者 这个季度收入看起来增长了，但利润没有同步变好——到底是哪些客户、项目、产品或部门在拉低毛利？

核心不是「有没有报表、能不能看图」——而是经营数据能否围绕当期管理问题被重新组织、被解释、被追问，并且让分析逻辑沉淀下来、下次还能复用。

老板问题 · 本质不是展示 · 是组织与沉淀

议题二 R0 + 本质判断。对老客户：不是「数进不了系统」，而是「有了数，怎么按管理问题看清楚、分析透、敢决策、能复用」。

Insight · Context

ERP 已有业务数据
经营分析还缺什么？

R0 追问层次企企 ERP 已提供经营分析中的缺口
哪些在拉低毛利？合同、项目、收入、成本、费用等已归集难按当期管理问题灵活组合呈现
为什么利润没跟上？各模块各有统计口径与标准报表口径、维度、分析目标靠人对齐，结论难复查
下次还会问类似问题吗？分析师个人经验可临时出结论分析过程难沉淀为团队可复用资产

瓶颈不在录入——数据已在系统中；缺口在围绕管理问题的组织、解释、追问与复用。

业务已归集 · 分析仍靠人拼口径

三行表分别对应 R0 的三个层次。财务/老板：固定报表答不了「这个季度为什么利润没跟上」。IT：拼数、口径、权限仍是痛点。

Insight · Practice

构建过程中的
实践认知

01Code 能力 · 交给 Agent 写

认知：LLM 代码能力已足够强，不必再堆配置模型

图表构建由 Agent 生成配置，而非构建配置模型；数据处理写代码完成；PaaS 平台脚本亦由 Agent 生产。

02边界与上下文 · 释放模型

认知：Harness 的价值在治理与连接，不在替模型套死流程

做好边界与上下文控制，充分释放模型能力——避免固化流程、把模型困在既定套路里。

03效果优先 · 质量飞轮

认知：软件 3.0 时代，效果取决于模型能力与工程构建

模型高频更新，工程需持续适配——效果观测与质量飞轮极其重要，上线不是终点。

04进入真实业务 · 而非新瓶装旧酒

坑：做成「更大号聊天框」、更复杂的固定报表，或低代码配置堆叠

解法：让 AI 进入真实业务过程——领域交互、数据切面、受控执行；关键动作人确认，能力与经验持续沉淀。

Code 生产 · 释放模型 · 质量飞轮 · 真实业务

口播重点：01 图表/数据/脚本都交给 Agent 写代码；02 边界上下文、勿固化困住模型；03 效果飞轮。04 收束产品形态，与 02 不重复——02 讲 Harness 原则，04 讲反模式。
